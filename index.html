Technically Not Technical

Your no-jargon starter kit — from setup to "wait, I built that?"

Welcome. If you're reading this, you're probably the person on the team who's been told to "just start using AI" without anyone explaining how. Or maybe you've been tinkering on your own and hit a wall. Either way — you're in the right place.

First things first — I'm not a developer. My work world is typically a blend of marketing and culture. And somehow I'm building dashboards, automations, and entire workflows with AI. I break things. Regularly. I ask Claude to do something and it confidently does the exact wrong thing. Then I fix it and somehow end up with something better than what I originally asked for. (Not sure if my please and thank yous to Claude actually work — but where I'm from, you say thank you even to the self-checkout machine.)

The gap between "non-technical" and "building real things" is way smaller than you might think. This guide is me paying it forward. It leans heavy on Claude because that's what I use — but the concepts apply no matter which AI you work with.

A few reminders:

  • Set a pace and give yourself grace. One section at a time.
  • The goal isn't efficiency — it's effectiveness. This isn't about doing the same work faster. It's about doing work that wasn't even possible before.
  • If I can figure this out, so can you. I had zero head start. You won't need one either.

Here's to building cool sh*t,

— Brandi...the Builder

Real talk: CoWork vs Code

There's a lot of hype around CoWork, and most of what it does you can also do in Claude Code through Visual Studio Code (VS Code). They both edit files, run tasks, and remember what you tell them.

What CoWork offers that Code doesn't:

  • Dispatch — assign Claude a task from your phone while you're away from your desk
  • Scheduled tasks — set something to run on a timer (though you can also do this with n8n or Code on the Web)

What Code offers that CoWork doesn't:

  • Full control — you see exactly what Claude is doing, every command, every file change

I spend most of my time in VS Code with Claude Code. The desktop app stays open for the occasional scheduled task and when I want to dispatch something from my phone.

How auto-memory works (CoWork and Code)

When you correct Claude — "no, we use Slack for that, not email" — it writes that down automatically. Between sessions, Claude reviews what it's written down, keeps what's relevant, and cleans up the rest. You don't manage this. It happens on its own.

  • Correct Claude when it gets something wrong — that's how it learns
  • Memory is scoped per project — what Claude learns in one project doesn't bleed into another
  • Check anytime — type /memory to see what Claude has remembered

Claude starts every session with zero memory of the last one. A CLAUDE.md file fixes that — it's a plain text file Claude reads automatically at the start of every session. Your tone, tools, preferences, and rules in one place. Set it up once and Claude already knows how you work before you say a word.

Where it lives and how to find it

Your personal CLAUDE.md lives at: ~/.claude/CLAUDE.md

That's a hidden folder in your home directory. Don't save copies elsewhere — Claude only reads from this location. To find it: press Cmd+Shift+G in Finder and paste the path above, or just ask Claude "show me my CLAUDE.md."

How to create it

Open Claude Code or CoWork and say "Create a personal CLAUDE.md for me" — then tell it about yourself. Claude will create the file in the right location automatically.

What to put in it

Put the most important rules first — Claude pays more attention to what it reads first.

  1. How you like to work (tone, format, level of detail)
  2. What Claude should never do
  3. Who you are and your role
  4. Tools and systems you use

Use bullet points, keep it short, and update it when Claude keeps repeating the same mistake. Treat it like a living doc.

Chat does not read CLAUDE.md files. A Chat Project fixes that — it gives Claude the same context every time you open it. Chat has two places for context, and you need both:

Settings > Profile

Your personal preferences — tone, style, pet peeves. Applies to every Chat conversation. Go to claude.ai/settings/general.

Projects

Context specific to one topic. Click Projects in the left sidebar, create a new one, and click Set Instructions. You can also upload files (brand guides, docs, reference material) that Claude can pull from in every conversation inside the project.

What to put in your Project instructions
  • Who you are and your role
  • What this project is for
  • Your audience and goals
  • What Claude should always do
  • What Claude should never do

Same rules as your CLAUDE.md — be specific, keep it short, update when something isn't working.

Chat Projects live in the cloud. CoWork Projects live on your computer — which means CoWork can read your local files, run scheduled tasks, and remember what it learned. Think of it as a desk drawer: instructions, files, memory, and recurring tasks all in one place for one job.

Three ways to create one
  • Start from scratch — build a new folder with instructions and files
  • Import from a Chat Project — bring over files and instructions you've already set up
  • Use an existing folder — point CoWork at a folder on your computer and add instructions

Term What it actually means

Slash Command A shortcut that triggers a Skill or Plugin action. Type /something and Claude knows exactly what to do. Think of it like a speed dial.

Why this matters: Slash commands are shortcuts you type directly into Claude. Instead of explaining what you want in a full sentence, you type a / and a keyword and Claude knows exactly what to do. They work in CoWork and Code.

The ones worth knowing
Command What it does When to use it
/init Scans your project folder and generates a CLAUDE.md automatically First time setting up a project in Code
/plan Puts Claude in read-only research mode — it investigates your project and produces a plan before doing anything Before a complex task where you want Claude to think first, not act
/compact Compresses the conversation to free up space in the context window When the conversation is getting long — do this at 50%, not when you're already at the limit

You don't need to memorize these. Just know they exist — and come back to this table when you need one.

About your context window: Every message takes up space. Once it's full, Claude starts losing track of what you said earlier — it didn't forget, it ran out of room. Run /compact at around 50% usage (don't wait until you're at the limit), and start a new conversation when you switch topics.

Why this matters: The quality of what Claude gives you is directly tied to the quality of what you ask. A good prompt is not about magic words — it's about being specific, giving context, and telling Claude what you actually want.

What makes a good prompt
  • Say who you are, what you're trying to do, and give Claude the context — audience, tone, format, length
  • Tell it what you don't want, not just what you do
  • Be specific: "write a 3-sentence summary for a non-technical audience" beats "summarize this"
  • If it gives you something close but not right, say so — Claude can iterate
Starter template — copy and adapt this
I am a [your role] at [type of company].

I need you to [specific task].

Audience: [who will read or use this]
Tone: [casual / professional / direct / friendly]
Format: [bullet points / paragraph / table / email]
Length: [short / under 200 words / one page]

Do not: [anything you want Claude to avoid]

Here is the relevant context: [paste anything Claude needs to know]

This is where Claude Code comes in — and before you panic, you don't need to know how to code. Claude Code is just Claude with the ability to work directly inside your project folders. Think of it like giving Claude a key to your filing cabinet instead of having to hand it one document at a time. If you can describe what you want done, you can build a Skill.

Tip: Before you build a Skill, drop a CLAUDE.md file into your project folder. Claude reads it automatically before starting any task. For complex builds, start with /plan so Claude thinks through the approach before it acts.

A Skill is a saved set of instructions Claude follows every time — write it once, use it forever.

When should you build one?
  • You've typed the same prompt 5+ times
  • You're manually fixing the same things every time
  • You want someone on your team to run it without knowing how to prompt

Advanced Terms
Term What it actually means
API (Application Programming Interface) A way for two apps to talk to each other. Think of it like a waiter — you place your order, they go to the kitchen, and bring back the result. You probably won't need this until you're building something custom. Everywhere, behind the scenes
SDK (Software Development Kit) A pre-packaged toolkit developers use to connect Claude to their own apps. Think of it like IKEA instructions — everything you need, in one box. You probably won't need this until you're building something from scratch. For developers building custom tools
MCP (Model Context Protocol) A standard way to connect Claude to tools you already use — like Slack, your calendar, or your CRM. Think of it like a universal adapter that lets Claude plug into your existing apps. CoWork and Code
Subagent A mini Claude spun up to handle one specific part of a bigger task. Think of it like assigning one person on your team to handle just the research while you handle the writing.
AGENTS.md A file that gives sub-agents shared context — like a team briefing doc. Your CLAUDE.md tells Claude how to work with you. AGENTS.md tells the agents Claude spins up what they all need to know: company context, positioning, who the team is, where things live. Think of CLAUDE.md as your personal onboarding doc, and AGENTS.md as the company handbook everyone gets on day one.Code
Hook An automatic behavior that fires when Claude does something — like formatting a file after every edit or assigning you to a pull request when it's created. You describe what you want and Claude sets it up.Code

Instead of using AI one task at a time, you build named agents that each own a lane of your work. They run on their own — scheduled tasks, triggers, background jobs — so you're not prompting every time.

Group agents by the lanes that match your work, not by individual tasks. Give each one a name so you can talk about them like teammates.

How to think about it
  • Each agent owns one lane — marketing, operations, admin, people and culture
  • Skills, scheduled tasks, and connected tools from the previous sections become the jobs each agent does
  • An AGENTS.md file gives all your agents shared context — like a team handbook
  • Start with the agent whose jobs are the most concrete and repeatable
