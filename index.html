Technically Not Technical

Your no-jargon starter kit — from setup to "wait, I built that?"

Welcome. If you're reading this, you're probably the person on the team who's been told to "just start using AI" without anyone explaining how. Or maybe you've been tinkering on your own and hit a wall. Either way — you're in the right place.

I'm not a developer — my lane is marketing and people and culture. And somehow I'm building dashboards, automations, and entire workflows with AI. I break things. Regularly. I ask Claude to do something and it confidently does the exact wrong thing. Then I fix it and somehow end up with something better than what I originally asked for.

And the vocabulary? "PR" used to mean press release or performance review. Now it means pull request. "Push" doesn't mean shove — it means sending your work to the cloud. The vocabulary will trip you up. That's why this guide has vocabulary sections built in.

But here's what I've learned: the gap between "non-technical" and "building real things" is way smaller than you might think. This guide is me paying it forward. It leans heavy on Claude because that's what I use — but the concepts apply no matter which AI you work with.

A few reminders:

  • Set a pace and give yourself grace. One section at a time.
  • The goal isn't efficiency — it's effectiveness. This isn't about doing the same work faster. It's about doing work that wasn't even possible before.
  • If I can figure this out, so can you. I had zero head start. You won't need one either.

Here's to building cool sh*t,

— Brandi...the Builder

Set Up Your Tools

'Do this once. Never think about it again.'

Claude doesn't share memory across tools. Each one needs its own setup — but once it's done, Claude already knows you before you say a word.

The simplest mental model
  • Mainly use Chat → set up Settings > Profile first, then create Projects for recurring topics
  • Use CoWork → create a CoWork Project for each area of work
  • Use Code → build your CLAUDE.md, it does both jobs in one file
  • Use all three → CLAUDE.md is your source of truth. Copy the relevant parts to Chat settings and Projects.
How auto-memory works (CoWork and Code)

When you correct Claude — "no, we use Slack for that, not email" — it writes that down automatically. Between sessions, Claude reviews what it's written down, keeps what's relevant, and cleans up the rest. You don't manage this. It happens on its own.

  • Correct Claude when it gets something wrong — that's how it learns
  • Memory is scoped per project — what Claude learns in one project doesn't bleed into another
  • Check anytime — type /memory to see what Claude has remembered

Why this matters: Claude starts every session with zero memory of the last one. Without this file, you're re-explaining yourself every time. Set it up once and Claude already knows how you work before you say a word.

A plain text file Claude reads automatically at the start of every session — your tone, tools, preferences, and rules in one place.

Where it lives

Your personal CLAUDE.md lives at: ~/.claude/CLAUDE.md

That ~ means your home folder. The .claude folder is hidden by default (the dot at the front hides it in Finder). This is the only location Claude reads from — do not save copies elsewhere (like your Desktop). If you edit a copy somewhere else, Claude won't see the changes.

How to find it
  • In Finder — press Cmd+Shift+G and paste ~/.claude/CLAUDE.md
  • In Typora — File > Open, paste the same path
  • In Claude Code or CoWork — just say "show me my CLAUDE.md" or "open my CLAUDE.md"
How to create it

The easiest way: open Claude Code or CoWork and say "Create a personal CLAUDE.md for me" — then tell it about yourself. Claude will create the file in the right location automatically. You can also use one of these two methods:

Two methods
Method How it works Best for
Setup prompt Tell Claude about yourself — who you are, how you work, what to avoid — and it writes the file for you Starting from scratch, personal CLAUDE.md, Chat and CoWork users
/init Claude Code scans the files already in your project folder and generates a CLAUDE.md based on what it finds Existing project folders with files in them (Claude Code only)
What makes a CLAUDE.md work
  • Put your most important rules at the top — Claude pays more attention to what it reads first (this is called primacy bias)
  • Use bullet points and short headings — walls of text get skimmed, just like they do by humans
  • Keep it under 500 lines — longer files get followed less consistently
  • Update it when Claude keeps repeating the same mistake — treat it like a living doc, not a one-time setup
  • Don't paste full documentation into it — give Claude summaries and tell it where to find the rest
What to put in it (in this order)

Put the rules that shape Claude's behavior at the top. Put reference info (people, links, tools) at the bottom. Claude pays more attention to what it reads first.

  1. How you like to work (tone, format, level of detail)
  2. What Claude should never do
  3. Context you always need from Claude
  4. Who you are and your role
  5. Key people and context Claude needs to know
  6. Tools and systems you use
  7. Where things live (links, repos, dashboards)

Why this matters: Chat does not read CLAUDE.md files. But if you use Chat regularly, you are probably re-explaining yourself every conversation. A Chat Project fixes that — it gives Claude the same context every time you open it.

Chat has two places for context — you need both:

Settings > Profile

Your personal preferences. Applies to every Chat conversation globally. Go to claude.ai/settings/general and fill in the personal preferences field. This is where your tone, style, and pet peeves live.

Projects

Context specific to one topic or project. Only applies inside that Project. Go to Claude Chat, click Projects in the left sidebar, create a new Project, and click Set Instructions.

What to put in your Project instructions
  • Who you are and your role
  • What this project is for
  • Your audience and goals
  • Tools and resources relevant to this project
  • What Claude should always do
  • What Claude should never do
What makes instructions work
  • Be specific — "respond in bullet points under 5 lines" beats "keep it short"
  • Front-load the most important rules
  • Keep it under 500 words — longer instructions get followed less consistently
  • Update them when Claude keeps getting something wrong — treat it like a living doc

You can also add files — upload documents, brand guides, or reference material directly to the Project and Claude can pull from them in every conversation inside it.

Tools to turn on per Project
  • Web search and Research — turn on for most projects
  • Extended thinking — good for complex strategic work, turns off automatically for quick tasks
  • Connectors — turn on the tools relevant to that project (Atlassian, Slack, Box, etc.)

Why this matters: Chat Projects live in the cloud. CoWork Projects live on your computer. That means CoWork can read your local files, run scheduled tasks on its own, and remember what it learned — all scoped to one project so nothing bleeds into another.

Think of a CoWork Project as a desk drawer — everything Claude needs for one job is in one place: instructions, files, memory, and recurring tasks.

Three ways to create a CoWork Project
  • Start from scratch — build a new folder with instructions and files from the ground up
  • Import from a Chat Project — search your existing Chat projects and bring their files and instructions over
  • Use an existing folder — point CoWork at a folder already on your computer, then add instructions and context
What each CoWork Project includes
  • Instructions — tone, formatting, rules that apply to every task inside this project
  • Context — link local folders, Chat projects, or URLs so Claude can reference them
  • Scheduled Tasks — recurring work Claude runs on its own, scoped to this project
  • Memory — Claude remembers what it learns inside this project (does not carry over to other projects)

Why this matters: Slash commands are shortcuts you type directly into Claude. Instead of explaining what you want in a full sentence, you type a / and a keyword and Claude knows exactly what to do. They work in CoWork and Code.

The ones worth knowing
Command What it does When to use it
/init Scans your project folder and generates a CLAUDE.md automatically First time setting up a project in Code
/plan Puts Claude in read-only research mode — it investigates your project and produces a plan before doing anything Before a complex task where you want Claude to think first, not act
/compact Compresses the conversation to free up space in the context window When the conversation is getting long — do this at 50%, not when you're already at the limit
/context Shows exactly how much of the context window is being used and by what When Claude starts forgetting things or you want to check how much room is left
/clear Resets the conversation completely — fresh start When you're switching to a completely different task
/model Switches which Claude model you're using mid-conversation When you want a faster (cheaper) model for simple tasks or a more powerful one for complex work
/rewind Undoes the last thing Claude did When Claude made a change you don't want — like an undo button
/permissions Lets you pre-approve commands so Claude doesn't have to ask every time When you're tired of clicking "allow" on the same action repeatedly
/doctor Runs a diagnostic check on your Claude Code installation When something isn't working and you're not sure why

You don't need to memorize these. Just know they exist — and come back to this table when you need one.

About your context window: Every message takes up space. Once it's full, Claude starts losing track of what you said earlier — it didn't forget, it ran out of room. Run /compact at around 50% usage (don't wait until you're at the limit), and start a new conversation when you switch topics.

Why this matters: The quality of what Claude gives you is directly tied to the quality of what you ask. A good prompt is not about magic words — it's about being specific, giving context, and telling Claude what you actually want.

What makes a good prompt
  • Say who you are and what you're trying to do
  • Give Claude the context it needs — audience, tone, format, length
  • Tell it what you don't want, not just what you do
  • Be specific: "write a 3-sentence summary for a non-technical audience" beats "summarize this"
  • If it gives you something close but not right, say so — Claude can iterate

A note on company Skills and where they work

This matters and is easy to get wrong.

  • Skills built in a Code marketplace (like the examples below) are available in Claude Code only — they don't automatically show up in CoWork or Chat
  • To use the same Skill in CoWork, it needs to be packaged and installed as a CoWork plugin
  • In Chat, the closest equivalent is adding those instructions to a Project with custom instructions

So if your company builds a brand voice Skill in Code and you want your whole team to use it regardless of which tool they're in, you'll need to set it up in both places.

What kinds of Skills are worth building?

Based on how Anthropic's own team uses them internally:

  • Writing & Content — tone, format, brand voice, templates
  • Business Process — recurring tasks you run the same way every time
  • Research & Analysis — how to pull, summarize, and report on specific information
  • Verification & Review — checking that output meets your standards before it goes out
Tips before you build
  • Start with a Gotchas section — document the mistakes Claude keeps making so it stops making them
  • Don't over-instruct — give Claude what it needs, not a rigid script
  • The description field is critical — it's how Claude decides when to trigger the Skill automatically

What "connected" actually looks like

When your tools are connected through MCP servers or connectors, Claude can reach across systems in a single command. Instead of you copying data between apps, Claude does it.

Example: "Research this prospect" could trigger Claude to:

  1. Pull their LinkedIn profile
  2. Check if they exist in your CRM
  3. Score them against your ideal customer profile
  4. Draft a personalized message
  5. Queue it for sending

One command. Five systems. No tab-switching.

Why this matters: Hooks let you set up automatic behaviors that fire when Claude does something — like running a formatter after editing a file, or assigning yourself to a PR the moment it's created. You configure them once, and they run every time without you thinking about it.

Hooks live in your settings.json file. You can set them globally (all projects) or per-project. Claude Code can set these up for you — just describe the behavior you want.

What it does: Automatically cleans up and organizes Claude Code's memory between sessions. Over time, Claude's memory folder can accumulate noise, contradictions, and stale notes. Auto Dream reviews all your past sessions, merges what's still relevant, and removes what's outdated — like your brain consolidating memories during sleep.

How it works: Runs in the background after 24+ hours have passed and 5+ sessions have occurred since the last consolidation. It only writes to memory files (read-only for your project code) and uses a lock file so it can't run twice at the same time.

Four phases: (1) Orient — reads current memory to understand what exists. (2) Gather signal — searches your session transcripts for corrections, decisions, and recurring themes. (3) Consolidate — merges new info, converts relative dates to absolute, resolves contradictions. (4) Prune and index — keeps the memory index under 200 lines, removes stale pointers.

How to check: Run /memory in Claude Code to see if Auto-dream is on. You can also trigger it manually by asking Claude to "dream" or "consolidate memories."
