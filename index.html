Technically Not Technical

A no-jargon starter kit — from setup to "wait, I built that?"

Welcome. If you're reading this, you might be the person on the team who's been told to "just start using AI" without anyone explaining how. Or maybe you've been tinkering on your own and hit a wall. Either way — you're in the right place.

First things first — I'm not a developer. My work world is typically a blend of marketing and culture. And yet I'm building dashboards, automations, and entire workflows with AI. I break things. Regularly. I ask Claude to do something and it confidently does the exact wrong thing. Then I fix it and somehow end up with something better than what I originally asked for. (Not sure if my please and thank yous to Claude actually work — but I've also been caught saying thank you to a self-checkout machine.)

I built this guide because the gap between "non-technical" and "building real things" is way smaller than you might think. It leans heavy on Claude — that's what I use — but the concepts apply no matter which AI you work with.

A few reminders:

  • Set a pace and give yourself grace. One section at a time.
  • The goal isn't efficiency — it's effectiveness. This isn't about doing the same work faster. It's about doing work that wasn't even possible before.
  • These tools change constantly. New features, new names, new ways of doing things. It can feel overwhelming. Focus on what's in front of you, not what just launched.
  • If I can figure this out, so can you. I didn't have a cheat sheet. Now you do.

Here's to building cool sh*t,

— Brandi...the Builder

Real talk: CoWork vs Code

There's a lot of hype around CoWork, and most of what it does you can also do in Claude Code. They both edit files, run tasks, and remember what you tell them.

What CoWork offers that Code doesn't:

  • Dispatch — assign Claude a task from your phone while you're away from your desk
  • Scheduled tasks — set something to run on a timer (though you can also do this with n8n)

What Code offers that CoWork doesn't:

  • Full control — you see exactly what Claude is doing, every command, every file change
  • Version control — built-in git means you can track changes, undo mistakes, and collaborate with your team

I spend most of my time in VS Code with Claude Code. The desktop app stays open for the occasional scheduled task.

On cost: Claude Pro at $20/month is enough to get started. If you're building daily, Max at $100-200/month gives you more room.

Here's the annoying part — Claude doesn't remember you everywhere. Each tool has its own brain, so you have to set up each one. The good news? Once you do, Claude already knows you before you say a word.

Claude starts every session with zero memory of the last one. Which is...not great. A CLAUDE.md file fixes that — it's a plain text file Claude reads automatically at the start of every session. Your tone, tools, preferences, and rules in one place. Set it up once and Claude already knows how you work before you say a word.

Where it lives and how to find it

Your personal CLAUDE.md lives at: ~/.claude/CLAUDE.md — that means it's in a hidden folder called .claude inside your home folder (the ~). The dot at the front makes it invisible in Finder.

Don't save copies elsewhere — Claude only reads from this location. To find it: press Cmd+Shift+G in Finder and paste the path above, or just ask Claude in your best Jerry Maguire voice "show me my CLAUDE.md."

How to create it

Open Claude (Chat, CoWork, or Code) and say "Create a personal CLAUDE.md for me" — then tell it about yourself. Claude will create the file in the right location automatically.

What to put in your CLAUDE.md

Put the most important rules first — Claude pays more attention to what it reads first.

  • How you like to work (tone, format, level of detail)
  • What Claude should never do
  • Who you are and your role
  • Tools and systems you use

Use bullet points, keep it short, and update it when Claude keeps repeating the same mistake. Treat it like a living doc.

So here's a catch — Chat doesn't read CLAUDE.md files. That means all the preferences you set up? Chat ignores them. A Chat Project is how you fix that. Chat has two places for context, and you need both:

Settings > Profile

Your personal preferences — tone, style, pet peeves. Applies to every Chat conversation. Go to claude.ai/settings/general.

Projects

Context specific to one topic. Click Projects in the left sidebar, create a new one, and click Set Instructions. You can also upload files (brand guides, docs, reference material) that Claude can pull from in every conversation inside the project.

What to put in your Project instructions
  • Who you are and your role
  • What this project is for
  • Your audience and goals
  • What Claude should always do
  • What Claude should never do

Same rules as your CLAUDE.md — be specific, keep it short, update when something isn't working.

Chat Projects live in the cloud. CoWork Projects live on your computer — which means CoWork can read your local files, run scheduled tasks, and remember what it learned. Think of it as a desk drawer: instructions, files, memory, and recurring tasks all in one place for one job.

Three ways to create one
  • Start from scratch — build a new folder with instructions and files
  • Import from a Chat Project — bring over files and instructions you've already set up
  • Use an existing folder — point CoWork at a folder on your computer and add instructions

CoWork Projects are desktop-only and don't sync across devices.

Type a / and a keyword and Claude just...does the thing. Slash commands are shortcuts built into Claude. They work in CoWork and in the terminal version of Claude Code. Heads up — if you're using the VS Code sidebar chat (like I usually do), most of these won't work there. You'd need to open a terminal session by typing claude.

The ones worth knowing
Command What it does When to use it
/init Scans your project folder and generates a CLAUDE.md automatically First time setting up a project in Code
/plan Puts Claude in read-only research mode — it investigates your project and produces a plan before doing anything Before a complex task where you want Claude to think first, not act
/compact Compresses the conversation to free up space in the context window When the conversation is getting long — do this at 50%, not when you're already at the limit
/context Shows exactly how much of the context window is being used and by what When Claude starts forgetting things or you want to check how much room is left
/clear Resets the conversation completely — fresh start When you're switching to a completely different task
/model Switches which Claude model you're using mid-conversation When you want to swap between faster or more powerful models
/rewind Undoes the last thing Claude did When Claude made a change you don't want
/btw Ask a side question without interrupting the current task When you have a squirrel moment and need to ask something without derailing what Claude is working on (terminal sessions only)
/memory Shows what Claude has remembered about you and this project When you want to see or manage what Claude knows

You don't need to memorize these. Just know they exist — and come back to this table when you need one.

About your context window: Every message takes up space. Once it's full, Claude starts losing track of what you said earlier — it didn't forget, it ran out of room. Run /compact at around 50% usage (don't wait until you're at the limit), and start a new conversation when you switch topics.
Your safety net: Press Esc to stop Claude mid-action. Press Esc twice to rewind — it restores your conversation and code to a previous state. Knowing you can undo anything makes you braver about trying things.

You know the phrase "garbage in, garbage out"? That's prompting. The more specific you are about what you want, the less time you spend fixing what you get back.

What makes a good prompt
  • Say who you are, what you're trying to do, and give Claude the context — audience, tone, format, length
  • Tell it what you don't want, not just what you do
  • Be specific: "write a 3-sentence summary for a non-technical audience" beats "summarize this"
  • If it gives you something close but not right, say so — Claude can iterate
Starter template — copy and adapt this
I am a [your role] at [type of company].

I need you to [specific task].

Audience: [who will read or use this]
Tone: [casual / professional / direct / friendly]
Format: [bullet points / paragraph / table / email]
Length: [short / under 200 words / one page]

Do not: [anything you want Claude to avoid]

Here is the relevant context: [paste anything Claude needs to know]
Pro tips
  • "Read it back to me." Before Claude starts building something complex, have a conversation first. Get aligned on the goal, then say "read it back to me" to confirm you're on the same page before it acts.
  • Change Claude's beliefs first. Don't jump straight to "write me a document." Start with a conversation. Share your thinking, your goals, your constraints. Get Claude to understand what you believe — then ask it to produce the output. The results are dramatically better.
  • Claude is the safest place to ask the questions you'd never ask out loud. "What does that term even mean?" "Am I thinking about this wrong?" No judgment. Ask away.
  • Tell Claude what "done" looks like. Before it starts, say "when you're done, I should be able to see X." Gives Claude a target instead of guessing what you want.
  • For big tasks, let Claude interview you first. Say "interview me about what I want before you start building." Claude will ask questions you haven't thought of — edge cases, tradeoffs, details you'd miss. Then it builds with a full picture.

This is where Claude Code comes in — and before you panic, you don't need to know how to code. Claude Code is just Claude with the ability to work directly inside your project folders. Think of it like giving Claude a key to your filing cabinet instead of having to hand it one document at a time. If you can describe what you want done, you can build a Skill.

A Skill is a saved set of instructions Claude follows every time — write it once, use it forever.

When should you build one?
  • You've typed the same prompt 5+ times
  • You're manually fixing the same things every time
  • You want someone on your team to run it without knowing how to prompt

Build What Didn't Exist Before

'Nobody asked you to build this. That's the point.'

Advanced Terms
Term What it actually means
API (Application Programming Interface) A way for two apps to talk to each other. Think of it like a waiter — you place your order, they go to the kitchen, and bring back the result. You probably won't need this until you're building something custom. Everywhere, behind the scenes
SDK (Software Development Kit) A pre-packaged toolkit developers use to connect Claude to their own apps. Think of it like a box of pre-cut building blocks — the pieces are ready, you just assemble them. You probably won't need this until you're building something from scratch. For developers building custom tools
MCP (Model Context Protocol) A standard way to connect Claude to tools you already use — like Slack, your calendar, or your CRM. Think of it like a universal adapter that lets Claude plug into your existing apps. CoWork and Code
Subagent A mini Claude spun up to handle one specific part of a bigger task. Think of it like assigning one person on your team to handle just the research while you handle the writing.
AGENTS.md A file that gives sub-agents shared context — like a team briefing doc. Your CLAUDE.md tells Claude how to work with you. AGENTS.md tells the agents Claude spins up what they all need to know: company context, positioning, who the team is, where things live. Think of CLAUDE.md as your personal onboarding doc, and AGENTS.md as the company handbook everyone gets on day one.Code
Hook An automatic behavior that fires when Claude does something — like formatting a file after every edit or assigning you to a pull request when it's created. You describe what you want and Claude sets it up.Code

Up until now, you've been asking Claude to do one thing at a time. This is where it gets fun — you build named agents that each handle a whole area of your work. Think of them like teammates who never call in sick.

Don't create an agent for every little task. Think about the big buckets — marketing, operations, admin, people stuff — and give each bucket an agent. Name them so you can talk about them like real people.

How to think about it
  • Each agent owns one lane — marketing, operations, admin, people and culture
  • Skills, scheduled tasks, and connected tools from the previous sections become the jobs each agent does
  • An AGENTS.md file gives all your agents shared context — like a team handbook
  • Start with the agent whose jobs are the most concrete and repeatable
